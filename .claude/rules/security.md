# Security

POC scope, so some controls are deferred. The non-negotiables below apply regardless of POC status — they're cheap to enforce now and expensive to retrofit.

## Secrets
- Never commit secrets. `.env.local` is gitignored; `.env.example` lists every key with a placeholder
- All secrets read via `lib/env.ts`, validated with Zod at startup. App refuses to boot if a required var is missing
- No secrets in client bundles. If a var name starts with `NEXT_PUBLIC_`, assume the world sees it
- Rotating a secret = rotate in Vercel AND revoke at the source. Document rotations in `docs/runbooks/`

## Logging — hard rules (apply even in POC)
Never log:
- Request bodies or response bodies
- Full URLs with query strings — strip query params before logging
- Headers (especially `authorization`, `cookie`, `x-api-key`)
- Tokens, API keys, session ids, password fields (even hashed)
- User PII beyond a user id
- Source code generated by the template engine — may contain user secrets
- Production stack traces with local variable values

Pino redaction in `lib/logger.ts`. Do not bypass it.

## Auth — POC stance
- POC runs with the `NoopAuthAdapter`. There is no real authentication.
- This is acceptable ONLY because the POC is deployed in a controlled environment (private deploy, IP allowlist, or VPN). Do not expose the POC URL publicly without at least basic-auth at the edge (Vercel password protection is fine).
- Every route that would need auth still calls `getCurrentUser()` from `lib/auth/` — preserving the seam is the whole point
- When the Okta or Entra adapter is wired up post-POC, the swap is one env var (`AUTH_PROVIDER=okta`) and a real implementation in `lib/auth/adapters/okta.ts`
- Future adapter implementations MUST validate ID tokens via the IdP's JWKS endpoint, not trust claims blindly. Resource Indicators (RFC 8707) where applicable.

## Input validation
- Every server action and API route validates input with Zod at the boundary
- After validation, trust the types inside
- Generated code is plain text in a ZIP — never executed server-side except inside a sandboxed CI boot test

## Output & rendering
- Never render user-supplied HTML
- Markdown via a strict allowlist renderer in `lib/markdown.ts`
- No `dangerouslySetInnerHTML` outside that renderer

## Dependency policy
- New runtime deps: license check (MIT / Apache-2.0 / BSD / ISC), maintenance check (>1 maintainer, released in last 12 months, >1k weekly downloads), one-line note in the relevant ADR
- `pnpm audit` runs in CI; high/critical = build fails
- Removal is easier than addition — be skeptical of each new dep

## Rate limiting
- Even in POC: rate-limit `/api/v1/generate` to prevent abuse from accidental loops (in-memory adapter is fine)
- Limits in `lib/rate-limit/`, keyed by the dev user id for the POC; will key by workspace once real auth lands

## Data handling
- User-submitted tool definitions are workspace-scoped and never leave our infra except as the user's own downloaded artifact
- POC artifacts in local filesystem (`.artifacts/`); a 7-day cleanup script runs in dev/CI
- When the R2 adapter is wired post-POC: 30-day expiry default

## Cookies & CORS
- All cookies: `httpOnly`, `secure`, `sameSite=lax` minimum (`strict` where possible)
- CORS for POC: same-origin only. Allowlist arrives when external integrations do.

## Generated code safety (template safety)
- Templates never hardcode secrets — all credentials use environment variables in the generated project
- The generator refuses to render if user-supplied tool descriptions match secret-shaped regex (AWS keys, JWTs, OpenAI sk-* keys, etc.)
- Generated `README.md` documents that any deployment of the output server should enforce auth at the transport layer (OAuth 2.1 + PKCE for Streamable HTTP), even though the POC outputs leave it as a TODO

## Reporting
- Vulnerabilities go to security@<domain>
- Issues opened with the `security` label are private until disclosed
- Dependabot alerts triaged within 3 business days
